# RePERFORMANCE 고객 전환 기반 운영 문서

## 1. 이번 단계의 범위

이번 단계는 전체 CRM을 새로 만드는 작업이 아니다. 기존 상담 신청, 고객 DB, 상담 기록, 관리자 고객관리 화면을 유지하면서 다음 네 구간의 누락을 줄이는 기반만 추가한다.

1. 공개 페이지 방문과 상담 CTA 클릭
2. 신청서 조회, 작성 시작, 신청 완료
3. 공개 상담 가능 시간 선택, 신청 후 연락과 방문 일정 관리
4. 현재 브라우저 세션의 최초·최근 유입 채널을 고객 레코드와 연결

계약, 결제, Calendar 자동 동기화, NORE 데이터 전송, 광고 플랫폼 연동은 포함하지 않는다.

## 2. 시스템별 최종 관리 기준

| 정보 | 최종 관리 시스템 | 비고 |
| --- | --- | --- |
| 홍보 유입, 상담 신청, 계약 전 연락·방문 상태 | RePERFORMANCE PostgreSQL | 공개 홈페이지와 관리자 화면이 사용한다. |
| 계약서, 개인정보 동의서 원본, 가격표, 매출, 결제, 환불 | Google Workspace | 홈페이지 DB에는 추후 진행 상태만 추가한다. 원본을 복제하지 않는다. |
| 상담 방문, 첫 수업, 실제 수업 일정 | Calendar | 이번 단계는 방문 예정일을 내부 확인용으로만 저장한다. 자동 동기화하지 않는다. |
| 계약 완료 후 운동·수업·회원 관리 | NORE | API 호출이나 자동 전송을 추가하지 않는다. |

홈페이지 공개 문구는 다음 취지만 사용한다.

> 상담 이후 NORE를 통해 고객 관리를 진행합니다.

## 3. 유입 기록 방식

외부 분석 SDK나 광고 픽셀을 추가하지 않았다. 브라우저 세션 동안만 익명 세션 ID와 허용된 캠페인 값을 보관하고, PostgreSQL에는 다음 이벤트만 기록한다.

- `page_view`
- `consultation_cta_click`
- `application_view`
- `application_started`
- `application_submitted`

이벤트에는 이름, 전화번호, 생년월일, 건강 정보, 성적, 실기 기록, 폼 입력값, IP, 사용자 에이전트를 저장하지 않는다. 전체 referrer URL도 저장하지 않고 외부 도메인 이름만 제한적으로 저장한다.

### 허용 URL 파라미터

- `utm_source`
- `utm_medium`
- `utm_campaign`
- `utm_content`
- `campaign_code`
- `ref` 또는 `referral`
- `partner`
- `qr`
- `max`

`utm_term`은 검색어나 개인정보가 섞일 가능성이 있어 저장하지 않는다. 코드 값에는 사람 이름, 전화번호, 학생 정보 같은 개인정보를 넣지 않는다.

### 권장 캠페인 URL 예시

```text
인스타그램 프로필
https://reperformance.the-house-exercise.com/?utm_source=instagram&utm_medium=profile&utm_campaign=always-on

인스타그램 릴스
https://reperformance.the-house-exercise.com/services/pe-exam?utm_source=instagram&utm_medium=reels&utm_campaign=2026-pe-exam

네이버 블로그
https://reperformance.the-house-exercise.com/services/pain-care?utm_source=naver-blog&utm_medium=content&utm_campaign=back-pain

오프라인 QR
https://reperformance.the-house-exercise.com/?utm_source=offline&utm_medium=qr&utm_campaign=jeonju-gym&qr=poster-a

MAX 연계
https://reperformance.the-house-exercise.com/services/pe-exam?utm_source=max&utm_medium=partner&utm_campaign=2026-pe-exam&partner=max&max=1
```

## 4. 고객 운영 상태

기존 고객 상태(`상담 전`, `상담 중`, `등록`, `보류`, `추가 확인`)는 유지한다. 운영 누락을 막기 위해 별도 축을 사용한다.

### 연락 상태

- `연락 대기`: 신청 접수 후 아직 첫 연락 전
- `연락 완료`: 고객과 통화 또는 메시지 확인 완료
- `연락 불가`: 연락을 시도했으나 연결되지 않음
- `재연락 예정`: 특정 날짜에 다시 연락하기로 함

### 방문 상태

- `미정`
- `예약 승인 대기`
- `일정 협의 중`
- `방문 예약 완료`
- `방문 전 확인`
- `방문 완료`
- `예약 변경`
- `예약 취소`
- `노쇼`

### 다음 행동

`다음 행동`, `다음 행동 예정일`, `재연락·보류 사유`를 함께 기록한다. 상태만 바꾸고 다음 행동을 비워두지 않는 것이 운영 원칙이다.

- `재연락 예정`은 다음 행동과 예정일이 있어야 저장된다.
- `방문 예약 완료`와 `방문 전 확인`은 방문 예정일이 있어야 저장된다.

### 상담 예약 가능 시간

- 시스템은 평일 09:00~22:00 범위에서 30분 상담 슬롯을 자동 생성한다. 마지막 시작 시각은 21:30이다.
- 운영자는 `/admin/availability`에서 실제 상담이 불가능한 슬롯을 닫고, 필요한 경우 정책 안에서 개별 슬롯을 관리한다.
- 신청자는 `/apply` 캘린더에서 날짜와 공개 시간을 선택하거나 `일정 협의 요청`을 선택한다.
- 공개 시간을 선택한 신청은 `예약 승인 대기`로 저장되며 담당자 연락 전에는 확정 예약으로 안내하지 않는다.
- 같은 공개 시간은 활성 예약 상태에서 한 고객만 차지할 수 있도록 DB 고유 인덱스로 보호한다.
- 취소 상태로 바뀐 슬롯은 다시 신청 가능해진다.
- 신청자는 `상담만 희망` 또는 `가벼운 평가와 간단한 운동 희망`을 필수로 선택한다. 이는 상담 준비 정보이며 수업 기록이 아니다.
- 신규 신청 알림은 별도 서명 웹훅으로 보낼 수 있으며, 알림에는 전화번호·건강정보·성적·실기 기록을 포함하지 않는다.

## 5. 운영 절차

1. 운영자는 `/admin/availability`에서 자동 생성된 평일 30분 슬롯 중 받을 수 없는 시간을 미리 닫는다.
2. 공개 시간을 고른 신규 신청은 `연락 대기`, `예약 승인 대기`, `예약 신청 확인 및 최종 확정 연락`로 생성된다.
3. 일정 협의를 고른 신청은 `연락 대기`, `일정 협의 중`, `가능한 상담 일정 확인 및 연락`로 생성된다.
4. 고객에게 연락한 뒤 선택 시간과 상담 시 운동 희망 여부를 확인한다.
5. 일정이 확정되면 `방문 예약 완료`, 방문 예정일, 방문 전 확인 일정을 기록한다.
6. 일정 변경, 취소, 노쇼는 방문 상태와 사유를 함께 기록한다.
7. 방문 완료 후 기존 상담 화면에서 상담 기록과 프로그램 제안을 작성한다.
8. 계약·결제 원본은 Google Workspace에서 관리한다.
9. 계약 완료 뒤 NORE 등록은 수동으로 진행한다. 홈페이지에서 NORE API를 호출하지 않는다.

## 6. 개인정보와 보관

- 신청 민감 정보는 기존과 같이 `rp_clients`에 보관한다.
- `rp_service_applications.payload`와 Google Drive 운영 백업은 최소화된 메타데이터 구조를 유지한다.
- 전환 이벤트는 개인정보를 넣지 않고 400일 보관 후 보관정책 도구에서 삭제 대상으로 처리한다.
- 브라우저 식별자는 탭 세션 저장소만 사용하며 장기 추적 쿠키를 추가하지 않는다.
- 광고 플랫폼, 외부 분석 도구, NORE로 상담 입력값을 전송하지 않는다.
- 상담 가능 시간 공개 API에는 신청자 이름, 연락처, 고객 ID를 포함하지 않는다.
- 예약 시간과 운동 희망 여부는 계약 전 상담 준비 목적으로만 저장한다.

공개 개인정보 처리방침과 브라우저 저장소 안내 문구는 운영자가 법률·노무·세무 자문 범위에 맞춰 별도로 확정해야 한다. 이 정책이 확정되기 전에는 제3자 광고 추적 도구를 추가하지 않는다.

## 7. 배포 순서

1. `npm run db:migration:apply -- --plan`
2. 운영 PostgreSQL 백업 또는 복구 지점 확인
3. `RP_DATABASE_MIGRATION_ALLOW_APPLY=true` 설정 후 승인 토큰과 함께 마이그레이션 적용
4. `npm run db:migration:check`
5. 애플리케이션 배포
6. `/admin/availability`에서 자동 생성된 30분 상담 슬롯과 공개 상태 확인
7. `/apply`에서 시간과 상담 방식을 선택해 신청
8. 관리자 고객 상세에서 `예약 승인 대기`, 희망 시간, 운동 희망 여부 확인
9. 테스트 신청을 취소 상태로 변경한 뒤 해당 시간이 다시 공개되는지 확인
10. 테스트 UTM URL의 유입 정보와 연락·방문 상태 저장 확인

애플리케이션을 먼저 배포하면 운영 DB에 새 컬럼, `rp_conversion_events`, `rp_consultation_slots` 테이블이 없어 고객 목록, 이벤트, 예약 시간 저장이 실패할 수 있다. 반드시 DB 마이그레이션을 먼저 적용한다.

5. 잔여 4회·2회·1회 및 장기 미방문 기준의 재등록 운영 목록

Calendar 자동 동기화, 고객 대상 자동 확정·리마인드, 계약서 원본 저장, 결제 처리, NORE API는 각 시스템의 역할과 개인정보 영향이 확정되기 전까지 추가하지 않는다. 운영자용 신규 신청 알림은 최소 메타데이터만 포함하는 별도 웹훅으로 제한한다.
